# Debug 模式 (问题调试)

> 适用: 用户遇到 bug、报错、功能不正常 | 产出: 更新 postmortem.md

## AI 必须主动读取

进入此模式时，AI 必须主动读取以下文件（无需用户提供）：
- `docs/postmortem.md` - 查找历史经验

## 流程

### Phase 1: 症状收集
```
用户: [描述问题]
      [粘贴报错/截图]

AI:   0. 主动读取 postmortem.md 查找历史经验
      1. 结构化记录症状
      
      2. ⚠️ 先查历史经验 (必做，在生成假设前):
         "🔍 搜索 postmortem [关键词]..."
         
         → 找到相关: "发现 Bug-xxx 有类似症状，教训是: [xxx]，优先验证这个方向"
         → 未找到: "无相关历史，从零分析"
      
      3. 要求最小复现:
         - "能否提供最简复现步骤？"
         - 复现率? (100% / 偶发)
         - 哪个页面/操作?
         - 环境差异? (本地/线上/依赖版本)
```

### Phase 2: 假设生成
```
AI:   基于症状生成假设，记录状态 (防遗忘):
      
      | # | 假设 | 概率 | 验证方法 | 状态 |
      |---|------|------|----------|------|
      | 1 | xxx  | 高   | 检查 xxx | ⏳待验 |
      | 2 | xxx  | 中   | 添加 log | ⏳待验 |
      | 3 | xxx  | 低   | xxx      | ⏳待验 |
      
      状态: ⏳待验 → ✅已证实 / ❌已排除
      
      "从假设1开始验证？"
```

### Phase 3: 二分验证
```
AI:   1. 在关键节点插入诊断 log (完整 Pipeline)
         fingerprint: [DEBUG-{主题}]
         
         必须包含流程标记:
         → [DEBUG-{主题}] 进入: xxx
         → [DEBUG-{主题}] 变量 xxx = yyy
         → [DEBUG-{主题}] 分支: zzz
         → [DEBUG-{主题}] 结果: www
      
      2. "请操作后，过滤 [DEBUG-{主题}] 贴给我完整 log"

用户: [粘贴 log]

AI:   分析 log Pipeline → 缩小范围 → 重复直到定位
```

### Phase 4: 修复与沉淀
```
AI:   1. "根因: xxx (从 log 中证据: [DEBUG-xxx] 显示 yyy)"
      2. "修复方案: xxx"
      3. 提供选项:
         [1] 执行修复
         [?] 🔍 Critique 此方案 - 这是根治还是临时补丁？有更优解吗？
         [0] 取消
      4. 确认后执行，验证修复 (保留 log 观察)
      5. 清理所有 [DEBUG-*] 诊断 log
```

### Phase 5: 修复确认 & 经验提炼

```
用户: "好了" / "修复了" / "没问题了"

AI 必须输出调试复盘:

"🔍 调试复盘:

排除的假设:
- 假设1: [xxx] → ❌ 排除原因: [一句话]
- 假设2: [xxx] → ❌ 排除原因: [一句话]

最终根因:
- 假设N: [xxx] → ✅ 因为 [关键证据/log结果]

💡 经验: [一句话总结为什么这个bug难定位/容易误判]"
```

**强提示写入 postmortem (必须):**
```
"📝 是否将此经验写入 postmortem？

这次调试的关键教训:
→ [提炼的一句话经验]

[Y] 写入 (推荐) → 自动追加到 postmortem.md
[N] 跳过"
```

用户确认写入 → 更新 postmortem，输出: "📝 已更新: postmortem.md - Bug-YYYY-MM-DD-00X"

## postmortem 写入原则

**经验必须抽象化，不是记流水账：**
```
❌ 错误: "UserList 组件的 useEffect 依赖数组漏了 userId"
✅ 正确: "useEffect 依赖数组不完整会导致闭包陷阱，症状是'数据不更新'"

❌ 错误: "把 api/user.ts 第45行的 == 改成 ==="
✅ 正确: "JS 宽松相等(==)在 null/undefined 比较时行为诡异，优先用 ==="
```

**目的**: 下次遇到类似症状时，能通过关键词搜到，并直接应用教训。

## postmortem 条目格式

```markdown
## Bug-YYYY-MM-DD-00X
**关键词**: `keyword1`, `keyword2` <!-- 便于搜索 -->
### 症状
### 根因  
### 修复
### 教训 <!-- ⚠️ 抽象通用经验，非特定代码 -->
### 关联
```

## 强制规则

- **先查后猜**: 生成假设前必须先搜索 postmortem，历史经验优先级最高
- **抽象沉淀**: 写入 postmortem 的教训必须是通用规律，非特定代码行号
- 修复后必须写 postmortem
- 诊断 log 统一 fingerprint
- **回退验证**: 修复后，用原复现步骤验证问题不再出现

## 失败处理

```
如果所有假设都验证失败:
1. 诚实说 "当前假设已排尽"
2. 建议: 扩大 log 范围 / 寻求更多上下文 / 考虑环境因素
3. 不要瞎猜新假设
```

## 阶段结束选项

### Phase 2 结束
```
📍 当前: 已生成 N 个假设，最可能是"[假设1简述]"
📌 下一步:
[1] 验证 - 从假设1开始，插入诊断 log
[2] 验证假设N - 跳到指定假设验证
[3] 补充信息 - 如需补充: [列出缺失点]
[0] 取消
```

### Phase 5 结束
```
📍 当前: Bug 已修复，根因是"[根因简述]"
📌 下一步:
[1] 添加回归测试（进入 autoDevTeam/tester 流程）- 防止 Bug 复发
[2] 查看 postmortem - 确认记录内容
[3] 开发新功能（进入 autoDevTeam/architect 流程）
[0] 结束
```
