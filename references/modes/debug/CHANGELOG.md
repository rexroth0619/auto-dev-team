# Debug 模式更新日志

## 2026-02-12: postmortem 写入时机修复

### 问题

**之前的流程**：
```
治疗方案执行完成
    ↓
即时验证（通过）
    ↓
⚠️ 自动写入 postmortem.md（不等用户确认）
    ↓
用户发现其实还有问题
```

**问题所在**：AI 在验证通过后就自动写入 postmortem，没有等待用户实际测试和确认 bug 确实修复了。

### 修复后的流程

```
治疗方案执行完成
    ↓
即时验证（通过）
    ↓
输出诊疗复盘 + 等待用户确认
    ↓
用户选择:
  [1] 确认修复 → 写入 postmortem.md ✅
  [2] 仍有问题 → 继续诊断
  [0] 暂不确认 → 稍后再测试
```

### 核心变更

#### Phase 5: 康复确认 & 病例归档

**之前**：
- ⛔ postmortem 归档是强制的，不是可选的
- 验证通过后立即执行，不等用户说"好了"
- 医师自动执行（不询问用户）

**现在**：
- ⭐ 只有在用户确认 bug 确实已修复后，才能写入 postmortem
- Step 5.1: 输出诊疗复盘 + 等待用户确认
- Step 5.2: 用户确认后才写入 postmortem.md

#### 新增三种结束情况

**情况 A：用户确认修复**
```
📌 下一步:
[1] 添加回归保护（进入 auto-dev-team/tester 流程）
[2] 开发新功能（进入 auto-dev-team/architect 流程）
[0] 结束
```

**情况 B：用户说仍有问题**
```
📌 下一步:
[1] 继续诊断 - 回到 Phase 2，重新生成鉴别诊断
[2] 详细描述问题 - 用户提供更多症状信息
[0] 暂停
```

**情况 C：用户暂不确认**
```
💡 提示: 测试完成后，可以继续本对话，回复"确认修复"或"仍有问题"
```

#### 强制规则更新

**之前**：
```
- ⛔ 治愈后自动归档病例（不询问，直接写入 postmortem.md）
```

**现在**：
```
- ⭐ 用户确认后归档: 用户确认 bug 已修复后，才写入 postmortem.md
```

#### 新增绝对禁止行为

```
⛔ 绝对禁止：
- 验证通过后不等用户确认就写入 postmortem
- 用户说"仍有问题"后仍然写入 postmortem
- 跳过用户确认环节
```

### 为什么这样改

1. **尊重用户验收权**：只有用户实际测试后，才能确认 bug 真正修复
2. **防止误判**：即时验证通过 ≠ bug 已修复（可能有边界情况、环境差异等）
3. **支持迭代修复**：用户说"仍有问题"时，可以继续诊断，不会误归档
4. **清晰责任边界**：AI 负责诊断和修复，用户负责最终验收

### 迁移指南

如果你的项目已经在使用旧版 Debug 模式：

1. **无需更改代码**：只是流程变化，不影响现有代码
2. **新对话生效**：新的 Debug 对话会自动使用新流程
3. **历史对话**：已归档的 postmortem 条目不受影响
