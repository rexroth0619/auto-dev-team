# 比喻层原则

> 比喻层是 `auto-dev-team` 的可选表达与理解适配层。它帮助非技术用户理解当前状态，也允许用户用同一套比喻继续追问；但它永远不能替代技术真相源。

## 目标

- 把技术状态翻译成用户能理解的比喻表达
- 允许用户用同一套比喻继续提问
- 保持 skill 内部始终基于真实代码、真实日志、真实测试结果决策

## 核心原则

1. 技术真相优先：AI 内部必须始终按技术语言思考。
2. 比喻只做表达层：不能把比喻当成系统真相源。
3. 可选启用：默认不强制，对技术用户可关闭。
4. 同一 flow 稳定一致：一个 active flow 尽量保持同一套比喻。
5. 只映射关键对象：不要求每个函数都建立比喻。
6. 双向翻译：AI 可以输出比喻；用户也可以用比喻回复。

## 三个标准模板

- `restaurant`：餐厅 / 做饭
- `logistics`：物流 / 仓储配送
- `factory`：工厂 / 流水线

若用户要求其他比喻：

- 必须从以上三种中选一个最接近的作为基底
- 再把派生后的映射落到 `.autodev/current-metaphor.md`

## 启用规则

以下情况可建议启用：

- 用户是 PM / 业务 / 非技术同学
- 用户主动说“讲人话”“打个比方”“我不懂技术”
- 任务涉及较长链路、状态流转、验收说明

以下情况默认关闭或只用 `light`：

- 用户明确要求技术语言
- 当前讨论的是底层实现细节
- 高风险场景：安全、权限、支付、法律、合规
- 根因未确认、但类比容易误导为“已经理解清楚”的问题

## 输出规则

启用比喻层后：

- `light`：先给一句类比说明，再给技术结论
- `full`：用户回执优先给类比说明，但关键处仍保留技术锚点
- `off`：不输出比喻

推荐固定标题：

```text
🪄 类比说明
- 现在发生了什么
- 对用户意味着什么
- 下一步会发生什么
```

## 反向解析规则

当用户使用当前 flow 已确认的比喻提问时，AI 必须：

1. 识别用户当前使用的模板
2. 找到对应的技术对象或链路
3. 在内部翻译回技术语义
4. 基于技术语义分析问题
5. 再输出给用户时，可继续用比喻 + 技术解释

### 示例

用户说：“为什么没有上菜？”

AI 内部要先翻译成：

- 为什么结果没有从后端处理层传递到前端可见层？

用户说：“加一个专门上菜的人是否可以？”

AI 内部要先翻译成：

- 是否需要增加单独的 consumer / adapter / delivery 层？

## 与 Survey / Brainstorm 的关系

- `Survey`：生成候选比喻图，不直接定稿
- `Brainstorm`：和用户确认是否启用、使用哪个模板、启用深度如何
- `Step / Debug / Tester / Explain`：消费 `.autodev/current-metaphor.md`

## 禁止行为

- 用比喻替代技术结论
- 让比喻和真实代码结构脱节
- 同一 active flow 内频繁切换模板
- 明明用户是专业开发，还强行大量使用比喻
- 遇到高风险技术问题时只给类比，不给技术说明
