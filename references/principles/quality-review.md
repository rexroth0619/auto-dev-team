# 质量复核

> 在确认“做的是这一题”之后，再检查“做得是不是足够稳、足够简洁、足够可验证”。

## 目标

- 检查代码结构是否合理
- 检查测试、观测、Blast Radius、GUI 验收是否对齐
- 检查是否引入过度设计、重复实现或遗漏对称路径
- 检查是否把“一类问题”错误地修成了“单点补丁”

## 何时触发

- Step 模式最后一步完成后
- FastTrack 完成后
- Debug 修复完成后

## 必查项

1. 结构：是否把新逻辑放进了正确模块。
2. 复杂度：是否引入 PM 未要求的设计。
3. 复用：是否错过已有模块或抽象机会。
4. 验证：后台自动测试、观测验证、GUI 验收是否与当前改动对应。
5. 风险：Blast Radius 是否覆盖真实改动。
6. 保留性：是否误删无关内容、误改对称路径。
7. 系统性：是否检查了类似模块 / 平行实现 / 对称路径；若只做单点修复，理由是否充分。
