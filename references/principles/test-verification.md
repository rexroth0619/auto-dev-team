# 测试编排协议

> canonical 测试文档。所有“后台自动测试 / 前端链路测试 / 手动验收 / 测试回执”都以本文件为准。

## 触发时机（强制）

- 任何代码更新
- 任何测试资产更新
- 任何可能影响运行行为的配置更新

## 核心规则

- **主 Agent 自己验证**，不把基础验证甩给测试 Subagent。
- **代码变更后默认先做后台自动测试**，不再把基础自动测试交给用户选择。
- **先影响分析，再组装场景矩阵，再执行测试**。
- **未跑 = 未验证**，不得用“应该没问题”替代测试结果。
- **命中用户链路风险时，AI 主动提示是否执行前端链路测试**。
- **命中大测试时，必须创建或更新 `.autodev/current-test.md`**。
- **业务规则不确定时必须先问**，未确认场景只能标记为“待业务确认”。
- **必须有证据**：命令、输出、截图说明、或明确的手动检查步骤。
- **失败重试最多 3 次**，仍失败就停止并报告。

## 四层测试

| 测试层 | 目标 | 默认动作 |
|--------|------|----------|
| 行为场景层 | 把 PM use case 翻译成可执行场景 | 生成场景矩阵，必要时落到 `.feature` 或 `current-test.md` |
| 后台自动测试层 | 验证逻辑、数据流、错误处理、兼容性 | 默认执行单元 / 集成 / 契约 / API smoke |
| 前端链路测试层 | 验证真实页面流程、跳转、会话、权限 | 命中触发器时主动提示用户是否执行 Playwright |
| 人工验收层 | 验证视觉、体感、外部系统、难自动化交互 | 输出具体检查步骤和预期结果 |

## 场景矩阵（默认补齐）

每个功能或修复至少从以下类型中选择适用项：

- `Happy`：正常路径
- `Negative`：错误输入、拒绝路径、失败提示
- `Zero / Empty`：空值、零值、未填写、无数据
- `Boundary`：阈值、极值、长度边界、页边界
- `Retry / Recovery`：重试、回滚、刷新、恢复
- `Permission / Role`：角色、权限、审批链
- `Manual only`：只能靠人工感知的场景

### AI 可以自动补的 case

- 通用负例
- 空值 / 零值 / 缺省值
- 边界值
- 错误链路
- 兼容旧调用或旧数据格式的常见回归 case

### 必须先问用户的 case

- 失败次数阈值、锁定策略、重试上限
- 角色权限矩阵
- 错误提示是否区分不同原因
- 超时、回滚、审批、幂等、并发业务规则
- 任何“对业务结果有影响，但仓库里没有明确约定”的规则

## 小测试 vs 大测试

| 类型 | 判断标准 | 产物 |
|------|----------|------|
| 小测试 | 单模块、低风险、非关键链路、可用少量自动测试说明白 | `🧾 测试回执` |
| 大测试 | 关键链路、跨模块、契约变更、业务规则不清、需要持续跟踪覆盖 | `.autodev/current-test.md` + `🧾 测试回执` |

### 大测试触发器（满足任一）

- 认证、支付、权限、审批、上传下载
- 多步表单、多页面跳转、会话状态、核心用户链路
- 外部系统联动
- API / 数据契约变更
- 跨模块改动
- 需要向 PM 追问业务规则

## 默认执行顺序

1. 做影响范围分析，明确改动点、直接调用方、直接消费方。
2. 组装场景矩阵，标出 `用户提出 / AI补充 / 待业务确认`。
3. 运行后台自动测试，至少覆盖：
   - 改动点
   - 1 个边界 case
   - 直接影响面中的关键调用方 / 消费方
4. 若项目已有相关测试，执行回归验证。
5. 判断是否命中前端链路测试触发器。
6. 命中大测试时，更新 `.autodev/current-test.md`。
7. 输出 `🧾 测试回执`。

## 后台自动测试（默认执行）

优先使用项目现有测试能力，常见包括：

- 单元测试：纯函数、规则计算、数据转换
- 集成测试：服务层、数据库、依赖协作
- 契约 / API 测试：接口返回、错误码、字段兼容
- 最小调用验证：CLI、`curl`、`node -e`、`python -c`

### 选择原则

- 优先选**最小但足够说明问题**的自动测试集。
- 修 bug 时，优先覆盖“原始复现路径 + 负例 + 边界例”。
- 功能尚未联调完成时，先测试当前已完成层级，不得假装整条链路已验证。

## 前端链路测试触发器

命中以下任一条件时，AI 必须主动提示用户是否执行 Playwright：

- 页面流程、核心表单、路由跳转
- 登录 / 注册 / 支付 / 权限 / 会话状态
- 上传下载、搜索筛选、关键用户操作链路
- 用户显式提到“前端验收”“全链路”“从用户角度”

### 提示模板

```text
⚠️ 这次改动触达用户链路，建议执行前端链路测试（Playwright）。
原因: [页面流程 / 跳转 / 会话 / 权限 / 核心表单]
是否现在执行？
```

### 结果约束

- 用户同意 → 执行 Playwright，并把场景、动作、断言、结果写入回执。
- 用户暂不执行 → 在回执和 `current-test.md` 中明确标记“未执行”。
- 当前步骤尚不具备前端联调条件 → 标记“暂不可执行，等待后续步骤接通”。

## 输出格式（必须）

```text
🧾 测试回执
测试等级: [小测试 / 大测试]
覆盖场景: [Happy / Negative / Boundary ...]
后台自动测试:
- 方式: [单测 / 集成 / 契约 / API smoke / CLI]
- 命令: [实际执行的命令]
- 结果: [通过 / 失败 / 无法执行]
- 证据: [关键输出]
前端链路测试: [未触发 / 建议执行 / 已执行 / 用户暂不执行 / 暂不可执行]
人工验收: [如需，给出步骤]
待业务确认: [无 / 列出问题]
剩余风险: [无 / 列出尚未覆盖的风险]
```

### 大测试附加要求

- `.autodev/current-test.md` 中必须同步更新：
  - 场景矩阵状态
  - 执行记录
  - 待业务确认问题
  - 剩余风险

## 失败处理

1. 先修代码，再重试验证。
2. 再检查测试方法是否本身有误。
3. 连续 3 次失败，停止推进并请求人工介入。
4. 若业务规则不明确，停止“继续补测”，改为向用户提问。

## 禁止行为

- 跳过后台自动测试直接说“已验证”。
- 只有结论，没有命令 / 输出 / 手动步骤。
- 命中大测试却不创建 / 更新 `current-test.md`。
- 把 `Playwright passed` 说成“真实用户上线一定没问题”。
- 把用户暂未确认的业务规则当成既定事实写进测试结论。
- 伪造测试输出或把验证责任甩给用户。
