# 自动化测试验证机制（One-Click Test）

> 目标：新功能完成或 Debug 修复完成后，**自动验证**，优先预发/生产一键测试，其次本地脚本，最后手动引导。

## 触发时机（强制）

- 新功能完成
- Debug 修复完成
- Hotfix 止血完成（可选，但建议）

## 缺失即补齐（强制）

当项目**不存在一键测试能力**时，必须先补齐**最小可用版本（MVP）**：

- 至少提供一个可被触发的测试入口（CLI 脚本或 API）
- 至少覆盖基础健康检查（如 health / DB 连接 / 配置校验）
- 必须写入 `docs/path.md`（入口、命令、环境）

**规则**：
- 不允许“先写功能、后补测试模块”
- 如果无法立即实现一键测试模块，必须解释原因，并降级到本地/手动测试

## 优先级（必须按顺序）

1. **预发/生产一键测试**（最高优先级）
2. **本地自动化测试脚本**
3. **手动测试引导**（不得跳过）

## 一键测试能力判定

### 有一键测试能力（满足任意一条）
- `docs/path.md` 中明确存在：`预发/生产 一键测试入口`（URL + 触发方式）
- `docs/project-map.md` 中存在一键测试模块/路由说明
- 管理后台已提供“一键测试”按钮

### 没有一键测试能力
必须明确提示：**项目缺少一键测试能力**，并提供以下两条路线：
1. **当前任务继续**：走本地脚本或手动测试
2. **新增能力**：建议为该项目补齐“一键测试模块”

## 自动执行规则

### 1) 预发/生产一键测试
```
AI:   1. 读取 docs/path.md，找到预发/生产一键测试入口
      2. 自动触发（如可执行）
      3. 输出测试结果摘要 + 如何查看完整日志
```

### 2) 本地自动化测试
```
AI:   1. 读取 docs/path.md 中测试命令（如 npm test / pytest）
      2. 能运行则直接执行并汇报结果
      3. 如果无法运行，解释原因并改为手动引导
```

### 3) 手动测试引导（兜底）
```
AI:   1. 输出清晰的手动测试步骤
      2. 每步包含预期结果
      3. 要求用户反馈结果
```

## 自动演进（强制）

每次新功能完成 / Debug 修复完成后，必须判断一键测试是否需要更新：

- 新增了新的核心流程 → 必须补充对应测试
- 新增了新的外部依赖 → 必须补充对应检查
- 新增了新的关键配置 → 必须补充配置校验

**输出**：
- `🧪 一键测试已更新: [新增覆盖点]`

## One-Click Test 标准模板（结构骨架）

> 这是**结构模板**，不是具体测试内容。每个项目按自身业务填写覆盖链路。

```

## 1) 测试入口
- 类型: [CLI / API / 后台按钮]

- 环境: [预发 / 生产 / 本地]

## 2) 覆盖链路清单（项目自定义）

## 3) 依赖与前置条件
- 账号/权限:
- 环境变量/密钥:
- 外部服务可用性:

## 4) 断言与证据
- 成功判定:
- 失败判定:
- 日志位置:

## 5) 失败处理
- 失败后动作:
- 复现方式:

- 测试数据清理:
- 回滚策略:
```

## 与项目文档的绑定（强制）

- `docs/path.md` 必须包含：
  - 一键测试入口（命令/URL/按钮）
  - 覆盖链路清单
  - 日志与证据位置
- `docs/path.md` 中需明确引用本模板（结构骨架）

## 项目级一键测试框架（跨项目固定）

> 这是“框架标准”，不是固定测试内容。每个项目必须基于此框架实现自己的一键测试模块。

### 分层结构（强制）

1. **基础巡检层（Smoke）**
   - 服务可用 / 配置可读 / 依赖可连通
   - 只读、低风险，生产/预发可跑
2. **业务链路层（Deep）**
   - 覆盖核心业务流程（可写入）
   - 仅预发可跑，必须带数据清理
3. **UI 自动化层（E2E）**
   - 使用 Puppeteer/Playwright 模拟真实用户操作
   - 仅预发可跑，必须输出可复现日志

### 能力组件（强制）

- **测试编排器**：统一执行顺序、超时控制、并发锁
- **测试用例库**：按业务场景拆分测试用例
- **数据生命周期**：测试数据创建、标记、清理
- **日志与证据**：每步有日志，失败可复现
- **权限与环境门禁**：生产仅允许 Smoke；Deep/E2E 必须预发

### 触发规则（强制）

- 新功能完成 / Debug 修复完成 → 必须决定执行哪一层
- 若项目已有 Deep/E2E 能力 → 默认执行
- 若缺失 → 必须补齐最小可用版本（先有 Smoke，再逐步扩展）

### 动态演进规则（强制）

每次功能完成时，必须评估并更新一键测试模块：
- 新增核心业务流程 → 增加 Deep 用例
- 新增 UI 交互 → 增加 E2E 用例
- 新增外部依赖 → 增加 Smoke 检查

**输出**：
- `🧪 一键测试已更新: [新增覆盖点]`

## 输出格式（必须）

```
🧪 自动化测试验证

选择路径: [预发一键测试 / 本地自动化 / 手动测试]
结果: [通过 / 失败 / 无法执行]
证据: [日志摘要 / 失败信息 / 执行原因]
```

## 禁止行为

- ❌ 直接跳过测试进入 commit
- ❌ 无法自动化时不做手动测试
- ❌ 只说“已测试”，不出示证据
