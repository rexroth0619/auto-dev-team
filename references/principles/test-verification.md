# 测试验证机制

> canonical 验证文档。所有“即时验证 / 回归 / 手动确认”都以本文件为准。

## 触发时机（强制）

- 任何代码更新
- 任何测试更新
- 任何可能影响运行行为的配置更新

## 核心规则

- **主 Agent 自己验证**，不把基础验证甩给测试 Subagent。
- **立刻执行**：写完代码当场验证，不积攒到最后。
- **先影响分析，再验证**。
- **必须有证据**：命令、输出、或明确的手动检查步骤。
- **失败重试最多 3 次**，仍失败就停止并报告。

## 三层验证

1. **即时验证（必做）**
   - 覆盖本次改动的核心逻辑。
   - 至少覆盖 1 个边界条件。

2. **回归验证（按需）**
   - 项目已有测试时，任务完成前运行相关测试。
   - 中到大改动优先做回归，不要只验改动点本身。

3. **手动确认（必要时）**
   - UI 视觉、外部系统、权限链路等无法自动化的场景。
   - 必须给出具体检查步骤，不能只说“请你测一下”。

## 验证方式选择

### 第一步：判断是否涉及 BDD 资产

若本次任务新增或修改 `.feature`、step definitions，必须先读取 `references/principles/bdd-testing.md`。

### 第二步：评估复杂度

| 维度 | 🟢 简单 | 🟡 中等 | 🔴 复杂 |
|------|--------|--------|--------|
| 改动量 | ≤2 文件且 ≤30 行 | 3-5 文件或 30-100 行 | >5 文件或 >100 行 |
| 功能类型 | 文案、样式、小修复 | 单模块逻辑改动 | 核心流程、跨模块、接口变更 |
| 影响范围 | 单点 | 单模块 | 多模块或对外行为 |

### 第三步：选验证手段

- **🟢 简单**：终端命令、临时脚本、最小可复现调用。
- **🟡 中等**：优先即时验证；若项目已有 BDD / 测试且场景清晰，可升级为自动测试。
- **🔴 复杂**：必须跑自动化验证；若项目已有 BDD，优先跑对应场景。

## 常见验证方式

- 终端直跑：CLI、`node -e`、`python -c`、`curl`
- 临时脚本：验证后立即删除
- 自动化测试：项目已有的单测、集成测试、BDD 场景
- 手动确认：给出逐步检查路径和预期结果

## 输出格式（必须）

```
🧪 即时验证
方式: [命令 / 脚本 / 自动化测试 / 手动确认]
结果: [通过 / 失败 / 无法执行]
证据: [关键输出、失败原因，或手动检查步骤]
```

## 失败处理

1. 先修代码，再重试验证。
2. 再检查验证方法是否本身有误。
3. 连续 3 次失败，停止推进并请求人工介入。

## 禁止行为

- 跳过验证直接建立存档或提交。
- 只有结论，没有证据。
- 明明是复杂改动，却只做“没报错”式验证。
- 伪造测试输出或把验证责任甩给用户。
