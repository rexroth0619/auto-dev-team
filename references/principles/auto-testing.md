# 自动化测试流程规范

> ⚕️ 每次手术后都要检查生命体征——不能做完手术不验证效果

## 概述

本规范定义了代码变更后的自动化测试完整流程，通过 Test Writer 和 Test Runner 两个 Subagent 实现。

## 核心组件

| 组件 | 位置 | 职责 |
|-----|-----|-----|
| Test Writer Subagent | `.cursor/agents/test-writer.md` | 生成测试代码 |
| Test Runner Subagent | `.cursor/agents/test-runner.md` | 执行测试、修复失败 |
| 影响范围分析 | 主 Agent 执行 | 确定测试覆盖范围 |

## 测试目录结构

```
项目根目录/
├── agent_test/                    # AI 生成的测试目录
│   ├── unit/                      # 单元测试
│   ├── integration/               # 集成测试
│   ├── e2e/                       # E2E 测试
│   └── screenshots/               # E2E 截图证据
├── docs/
│   └── path.md                    # 包含测试相关路径
```

## 触发时机与策略

### Step 模式（逐步确认）

```
每个 Step 完成后:

Step N 完成
    ↓
影响范围分析（主 Agent）
    ↓
调用 /test-writer
    ├── 输入：需求 + 代码 + 影响范围
    └── 输出：测试代码 → agent_test/
    ↓
调用 /test-runner
    ├── 运行测试
    ├── 失败则修复（最多 3 次）
    └── 输出：测试报告
    ↓
调用 /critique（已有机制）
    ↓
输出给 PM
    ↓
等待确认 → 继续 Step N+1
```

### 信任模式（连续执行）

```
信任模式（只在最后做合集测试）:

执行 Step 1
    ↓
执行 Step 2
    ↓
...
    ↓
执行 Step N
    ↓
汇总所有变更
    ↓
合并影响范围分析
    ↓
调用 /test-writer（合集模式）
    └── 为所有变更生成测试
    ↓
调用 /test-runner
    ↓
调用 /critique
    ↓
输出给 PM
```

### Debug 模式（修复后）

```
Bug 修复完成后:

修复代码
    ↓
影响范围分析
    ↓
调用 /test-writer
    ├── 针对 bug 场景的测试
    └── 影响范围的回归测试
    ↓
调用 /test-runner
    ↓
输出给 PM
```

### Hotfix 模式（紧急修复）

```
Hotfix 完成后（最小化测试）:

修复代码
    ↓
只测试核心路径（Smoke Test）
    └── 不做完整影响范围测试
    ↓
输出给 PM
    ↓
⚠️ 提醒：后续需要补充完整测试
```

## 测试框架自动安装

当项目没有测试框架时，Test Runner 自动安装：

| 项目类型 | 检测方式 | 默认框架 | 安装命令 |
|---------|---------|---------|---------|
| Node.js/TS | package.json | vitest | `npm install -D vitest` |
| React/Vue | package.json + 框架依赖 | vitest + @testing-library | `npm install -D vitest @testing-library/react` |
| 需要 E2E | 有 UI 组件 | Puppeteer | `npm install -D puppeteer` |
| Python | requirements.txt | pytest | `pip install pytest` |
| Go | go.mod | 内置 testing | 无需安装 |
| Java Maven | pom.xml | JUnit 5 | 添加 pom 依赖 |
| Java Gradle | build.gradle | JUnit 5 | 添加 gradle 依赖 |

## 失败处理机制

### 重试策略（最多 3 次）

```
第 1 次失败:
├── 分析错误信息
├── 判断：代码 bug vs 测试 bug
├── 修复问题
└── 重新运行

第 2 次失败:
├── 重新分析（换个角度）
├── 之前修代码 → 这次检查测试
├── 之前修测试 → 这次检查代码
└── 重新运行

第 3 次失败:
├── 停止重试
├── 输出详细失败报告
└── 请求人工介入
```

### 人工介入触发条件

| 情况 | 处理 |
|-----|-----|
| 3 次重试仍失败 | 必须人工介入 |
| 无法判断代码/测试 bug | 询问用户 |
| 环境问题（数据库/网络） | 提示用户检查环境 |
| 依赖缺失 | 提示安装依赖 |

## 测试代码生命周期

### 持久化策略

| 测试类型 | 默认策略 | 原因 |
|---------|---------|-----|
| 核心流程集成测试 | 🔴 保留 | 回归测试价值高 |
| E2E 测试 | 🔴 保留 | 完整流程验证 |
| 边界条件单测 | 🟡 可选 | 根据重要性决定 |
| 临时验证单测 | 🟢 删除 | 价值低，避免维护负担 |

### 清理时机

```
测试全部通过后:
1. 读取 test-writer 的持久化建议
2. 保留标记为"保留"的测试
3. 删除标记为"可删"的测试
4. 输出清理报告
```

## 与现有机制的整合

### 与 test-verification.md 的关系

`test-verification.md` 定义了"必须验证"的原则。

本文件（`auto-testing.md`）定义了具体实现：
- 如何生成测试（Test Writer）
- 如何运行测试（Test Runner）
- 失败如何处理

### 与 Critique Subagent 的关系

```
执行顺序:
1. 代码变更完成
2. 影响范围分析
3. Test Writer 生成测试
4. Test Runner 运行测试
5. Critique 审查方案（在测试通过后）
6. 输出给用户
```

测试在 Critique 之前，因为：
- 测试验证"实现是否正确"
- Critique 审查"方案是否合理"
- 实现都错了，审查方案没意义

### 与 incremental-testable.md 的关系

`incremental-testable.md` 要求每步可测。

本机制确保每步实际被测试：
- 之前：每步"可以测"（人工验证）
- 现在：每步"被测了"（自动测试）

## 禁止行为

- ❌ 跳过测试直接 commit（除非 Hotfix 紧急情况）
- ❌ 测试失败但报告成功
- ❌ 不执行影响范围分析
- ❌ 超过 3 次重试仍继续
- ❌ 不输出业务语言的测试报告
- ❌ 信任模式下每步都测（应该只测合集）
