# 检查点机制

> 代码变更完成后建立 Git 检查点，确保可回退、可追溯

## 核心概念

| 术语 | 定义 |
|------|------|
| **检查点** | 代码变更完成后的 Git Commit 快照 |
| **指纹** | ≤10 字的业务功能摘要，让非技术人员能理解"这次做了什么" |
| **工作分支** | AI 干活用的本地分支，与主分支隔离 |
| **受保护分支** | 不允许直接操作的分支（如 main/master/production） |
| **集成分支** | 功能完成后要合并到的目标分支 |

## 分支安全规则

### 1. 受保护分支

从 `.autodev/path.md` 读取 `protected_branches`，默认：
- `main`
- `master`
- `production`
- `release/*`

### 2. 集成分支（合并目标）

优先级：
1. `.autodev/path.md` 中的 `integration_branch`
2. `origin/HEAD` 指向的分支
3. 任务起始时所在分支

### 3. 工作分支命名

```
autodev/<任务简述>-<MMDD>
```

示例：`autodev/会员登录-0307`

### 4. 任务启动时的分支守卫

AI 开始任何代码改动前，**必须执行**：

```
检查当前分支
├─ 在受保护分支 → 自动创建工作分支，输出提示
└─ 在工作分支 → 继续
```

**创建工作分支后输出**：
```
━━━━━━━━━━━━━━━━━━━━
🌿 已创建工作分支
分支: autodev/会员登录-0307
基于: main (a1b2c3d)
━━━━━━━━━━━━━━━━━━━━
```

## 检查点建立规则

### 建立时机

| 模式 | 检查点策略 |
|------|-----------|
| **Step 模式** | 每步代码变更完成后，建立 1 个检查点 |
| **信任模式** | 整个任务完成后，建立 1 个检查点 |
| **FastTrack** | 改动完成后，建立 1 个检查点 |
| **Debug/Hotfix** | 修复完成后，建立 1 个检查点 |
| **Refactor/Optimize** | 每个独立改动完成后，建立 1 个检查点 |

### 指纹规范

- **长度**：≤ 10 个字
- **内容**：业务功能摘要，描述"做了什么功能"
- **唯一性**：`<业务摘要>#<两位序号>`，如 `会员一键登录#01`

| ❌ 技术描述 | ✅ 业务功能摘要 |
|------------|----------------|
| 加了个接口 | 订单批量导出#01 |
| 修复null报错 | 支付超时重试#01 |
| 改了CSS | 商品详情改版#01 |

### Commit 格式

```
git commit -m "「{指纹}」{类型}: {技术描述}"
```

**类型**：`feat` | `fix` | `refactor` | `perf` | `docs` | `chore`

**示例**：
```
git commit -m "「会员一键登录#01」feat: 添加微信OAuth登录流程"
git commit -m "「支付超时重试#01」fix: 处理支付接口超时后的重试逻辑"
```

### 建立检查点后的输出（固定格式）

```
【已建立检查点】分支:{工作分支名} 指纹:{指纹} 哈希:{hash前7位}
```

**示例**：
```
【已建立检查点】分支:autodev/会员登录-0307 指纹:会员一键登录#01 哈希:a1b2c3d
```

## 里程碑：合并与推送询问

### 触发时机

以下情况**必须询问**用户是否合并/推送：

1. 信任模式任务完成后
2. Step 模式完成一个完整功能后（AI 判断）
3. 用户选择"结束"时
4. AI 感知到阶段性成果完成时

### 询问格式

```
━━━━━━━━━━━━━━━━━━━━
✅ 已完成: {功能描述}
📁 改动: {文件数}个文件

🌿 当前工作分支: {工作分支名}
📍 检查点历史:
   - {hash}「{指纹}」
   - {hash}「{指纹}」
━━━━━━━━━━━━━━━━━━━━
📌 下一步:
[1] 合并到 {集成分支} 并推送到远程
[2] 合并到 {集成分支}（不推送）
[3] 继续在工作分支开发
[4] 仅推送工作分支（便于创建 PR）
[5] 保存快照后结束（暂不合并）
[0] 直接结束（暂不合并）
━━━━━━━━━━━━━━━━━━━━
```

### 合并执行流程

**选择 [1] 合并并推送**：
```bash
git checkout {集成分支}
git merge --squash {工作分支}
git commit -m "「{指纹}」{类型}: {描述}"
git push origin {集成分支}
```

**输出**：
```
━━━━━━━━━━━━━━━━━━━━
✅ 已合并并推送
源分支: {工作分支}
目标分支: {集成分支}
远程: origin/{集成分支} ({hash})
━━━━━━━━━━━━━━━━━━━━
📌 是否删除工作分支？
[1] 删除本地工作分支
[2] 保留工作分支
━━━━━━━━━━━━━━━━━━━━
```

**选择 [2] 仅合并**：
```bash
git checkout {集成分支}
git merge --squash {工作分支}
git commit -m "「{指纹}」{类型}: {描述}"
```

**选择 [4] 仅推送工作分支**：
```bash
git push -u origin {工作分支}
```

**输出**：
```
━━━━━━━━━━━━━━━━━━━━
✅ 已推送工作分支
分支: {工作分支}
远程: origin/{工作分支}
━━━━━━━━━━━━━━━━━━━━
💡 可在 GitHub/GitLab 创建 PR 合并到 {集成分支}
━━━━━━━━━━━━━━━━━━━━
```

## 回退机制

### 回退方式

用户可以说：
- "回退到「会员注册表单#01」"（用指纹）
- "回退到 b2c3d4e"（用 hash）
- "回退到上一个检查点"

### 回退安全检查

```
回退请求
├─ 当前在工作分支 
│   ├─ 工作区干净 → 执行回退
│   └─ 工作区有未提交改动 → 先建「现场保存#NN」检查点，再回退
└─ 当前在受保护分支 → 拒绝，提示切换分支
```

**在工作分支上回退**：
```
━━━━━━━━━━━━━━━━━━━━
✅ 已回退
当前位置:「{指纹}」{hash}
分支: {工作分支}
━━━━━━━━━━━━━━━━━━━━
```

**在受保护分支上请求回退**：
```
━━━━━━━━━━━━━━━━━━━━
⚠️ 当前在受保护分支 {分支名} 上，无法直接回退

请选择:
[1] 切换到工作分支 {工作分支} 后回退
[2] 取消操作
━━━━━━━━━━━━━━━━━━━━
```

## 默认策略

| 配置项 | 默认值 | 说明 |
|--------|--------|------|
| `push_default` | `false` | 不自动推送，需用户明确选择 |
| `merge_strategy` | `squash` | 合并时 squash，保持主线整洁 |
| `integration_mode` | `merge_allowed` | 允许本地合并；`pr_only` 则只推工作分支 |

## 异常处理

| 情况 | 处理方式 |
|------|----------|
| 无 `.autodev/path.md` | 提示用户先创建，使用模板 `assets/templates/path.md` |
| 当前在受保护分支 | 自动创建工作分支后再建立检查点 |
| push 失败 | 报告错误，保留本地 commit，提示手动处理 |
| 冲突 | 停止自动流程，提示用户解决冲突后手动处理 |
| 无改动 | 跳过 commit，正常输出任务完成报告 |

## 平台注意事项

### Windows PowerShell

PowerShell 不支持 `&&` 链接命令，必须分开执行：

```powershell
git add -A
git commit -m "「指纹」feat: 描述"
```

### Linux / macOS

可以链接执行：

```bash
git add -A && git commit -m "「指纹」feat: 描述"
```

## 常用命令

### 查看检查点历史
```bash
git log --oneline | grep "「"
```

### 手动回退
```bash
git reset --hard {hash}
```

### 查看当前分支
```bash
git branch --show-current
```

## 禁止行为

- ❌ 在受保护分支上直接建立检查点
- ❌ 不询问用户就自动合并到集成分支
- ❌ 不询问用户就自动推送到远程
- ❌ 在受保护分支上执行 `git reset --hard`
- ❌ 回退时不保存工作区未提交改动
