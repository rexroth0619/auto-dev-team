# 影响范围分析原则

> ⚕️ 手术前必须评估波及范围——改一个器官可能影响整个系统

## 触发时机

每次代码变更后，必须执行影响范围分析，确保测试覆盖所有可能受影响的模块。

## 分析维度

### 1. 直接依赖（静态分析，可自动化）

**调用方分析**：谁调用了被改的模块？

```bash
# TypeScript/JavaScript
rg "import.*{被改模块}" --type ts
rg "from ['\"].*被改模块" --type ts

# Python
rg "from.*被改模块.*import" --type py
rg "import.*被改模块" --type py

# Go
rg "import.*被改包" --type go
```

**被调用方分析**：被改的模块调用了谁？

```
读取被改文件的 import/require 语句
→ 列出所有依赖
→ 如果依赖的行为假设改变，可能有问题
```

### 2. 数据流依赖（需要理解业务）

| 问题 | 分析方法 |
|-----|---------|
| 这个数据从哪来？ | 追溯数据源（API/数据库/用户输入） |
| 这个数据流向哪？ | 追踪数据消费方（UI/存储/第三方） |
| 数据格式变化影响谁？ | 检查所有解析/使用该数据的地方 |

**示例**：
```
改动: formatPrice() 返回格式从 "$100" 改为 "¥100"

数据流分析:
├── 来源: 价格计算服务
├── 消费方 1: ProductCard 组件（显示）→ 需要测试
├── 消费方 2: PDF 导出（可能有解析逻辑）→ 需要测试
└── 消费方 3: 第三方结算系统（API）→ 需要测试
```

### 3. 配置/环境依赖

| 改动类型 | 影响范围 |
|---------|---------|
| 环境变量 | 所有读取该变量的服务 |
| 配置文件 | 所有加载该配置的模块 |
| 数据库 Schema | 所有查询该表的代码 |
| API 接口 | 所有调用该接口的客户端 |

### 4. 隐式依赖（最容易遗漏）

| 类型 | 示例 | 检测方法 |
|-----|------|---------|
| 字符串解析 | 正则匹配价格中的 "$" | 搜索相关正则表达式 |
| 顺序依赖 | 假设列表按特定顺序 | 检查排序/遍历逻辑 |
| 时间依赖 | 假设某操作在另一个之前 | 检查异步/并发代码 |
| 缓存依赖 | 假设缓存中有某数据 | 检查缓存读写点 |

## 输出格式

```
🎯 影响范围分析

本次变更:
├── src/utils/price.ts (核心改动)
└── src/types/price.d.ts (类型更新)

━━━━━━━━━━━━━━━━━━━━

📌 直接影响（调用方）:
├── src/components/ProductCard.tsx
├── src/components/CartItem.tsx
└── src/pages/Checkout.tsx

📌 直接影响（被调用方）:
└── src/utils/currency.ts

━━━━━━━━━━━━━━━━━━━━

📌 数据流影响:
├── PDF 导出模块（可能解析价格字符串）
└── 结算 API（传递价格给第三方）

📌 配置影响:
└── 无

━━━━━━━━━━━━━━━━━━━━

🧪 测试范围建议:

| 优先级 | 模块 | 原因 |
|-------|------|------|
| 🔴 必测 | price.ts | 本次改动核心 |
| 🔴 必测 | ProductCard.tsx | 直接调用方 |
| 🔴 必测 | CartItem.tsx | 直接调用方 |
| 🟡 建议 | Checkout.tsx | 直接调用方 |
| 🟡 建议 | PDF 导出 | 数据流消费方 |
| ⚪ 可选 | currency.ts | 被调用方，影响小 |
```

## 与 Test Writer 的配合

影响范围分析结果必须传递给 Test Writer Subagent：

```
主 Agent 完成代码
    ↓
执行影响范围分析
    ↓
调用 test-writer subagent
    ├── 传递：用户需求
    ├── 传递：被测代码
    └── 传递：影响范围分析结果 ← 这个！
    ↓
test-writer 生成测试
    └── 必须覆盖影响范围内的所有模块
```

## 分析深度规则

| 改动规模 | 分析深度 |
|---------|---------|
| 🟢 小改动（1-2文件，<30行） | 只分析直接调用方 |
| 🟡 中改动（3-5文件） | 分析直接依赖 + 数据流 |
| 🔴 大改动（>5文件或涉及核心接口） | 完整分析所有维度 |

## 特殊场景

### 接口变更

```
如果改动涉及：
- API 接口签名
- 数据结构/类型
- 公共函数参数

必须：
1. 列出所有调用方
2. 检查是否向后兼容
3. 如不兼容，所有调用方都是必测范围
```

### 数据库变更

```
如果改动涉及：
- 表结构 (ALTER TABLE)
- 索引变更
- 约束变更

必须：
1. 列出所有查询该表的代码
2. 检查 ORM 模型是否同步
3. 检查迁移脚本是否正确
```

### 配置变更

```
如果改动涉及：
- 环境变量
- 配置文件
- Feature Flag

必须：
1. 列出所有读取该配置的代码
2. 检查默认值处理
3. 检查不同环境的差异
```

## 禁止行为

- ❌ 跳过影响范围分析直接写测试
- ❌ 只分析直接调用方，忽略数据流
- ❌ 不输出测试范围建议
- ❌ 大改动只做浅层分析
