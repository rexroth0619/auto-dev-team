# 即时验证 Protocol

> 目的：**快、清晰、可复现**。

## 必做清单

1. 核心逻辑：本次改动的主功能
2. 边界条件：至少 1 个边界输入
3. 调用方：有明显调用方则顺手验证

## 方法选择

- 终端直跑（优先）
- 临时脚本（验证后删除）
- 无法自动验证 → 明确给用户检查步骤

## 通过标准

- 必须有预期值
- 必须对比实际输出
- 不能只看“没报错”

## 失败处理

1. 优先怀疑代码，再检查验证逻辑
2. 修复后重试，最多 3 次
3. 仍失败 → 请求人工介入
# 即时验证协议（主 Agent）

> 目标：在当前上下文，快速证明“改动有效且没破坏关键路径”。

## 必做验证点

1. 核心逻辑（本次改动的主要功能）
2. 至少 1 个边界条件（空值/极值/异常输入）
3. 若有直接调用方，至少验证 1 个

## 验证方式选择（优先快）

- 纯函数/工具：`node -e` / `python - <<'PY'`
- API：`curl` / 临时请求脚本
- 文件/配置：直接读取或检查输出文件
- UI/体验：给出明确手动验证步骤

## 临时代码规则

- 优先**一行命令**验证
- 需要临时脚本时：用 `.tmp-verify.*`，验证后删除
- 禁止把验证代码留在正式文件

## 失败处理（最多 3 次）

1. 先假设代码问题 → 修复 → 重试
2. 再检查验证逻辑是否错误
3. 第 3 次失败 → 停止并请求用户介入

## 输出格式

```
🧪 即时验证: [通过/失败]
证据: [命令/输出摘要]
```
