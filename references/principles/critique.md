# 自动会诊机制

> ⚕️ 每个方案都需要第二意见——主治医生自己审自己有 bias

## 机制说明

**问题**：新对话后 AI 丢失上下文，可能提出过度设计或临时补丁方案。主 Agent 自己审自己有 confirmation bias。更危险的是：用户需求本身可能不合理、有歧义、或缺少关键信息，主 Agent 往往直接执行而不质疑。

**解决方案**：每次输出方案后，**自动调用独立的 Critique Subagent** 进行审查。Subagent 不仅审查方案本身，还要回溯审查用户的原始需求。

## 触发时机（强制自动，不再是可选项）

- 提出新功能开发方案后
- 提出 Debug 修复方案后
- 提出重构/优化方案后
- 任何涉及代码改动的方案输出后

## 执行流程

```
1. 主 Agent 生成方案
2. 自动调用 Critique Subagent（不等用户选择）
   - 传递：【用户原始需求】+ 方案内容、诊断结论、理由
   - 不传递：主 Agent 的思考过程（避免 bias 传递）
3. Subagent 执行两阶段审查：
   ┌─────────────────────────────────────────┐
   │ 阶段 A: 需求澄清检查（Clarification）    │
   │ → 用户需求是否合理？是否有歧义？         │
   │ → 是否缺少关键信息？                     │
   │ → 是否需要用户二次确认？                 │
   │                                         │
   │ ⚠️ 如有澄清问题，立即暂停，先问用户      │
   └─────────────────────────────────────────┘
   ┌─────────────────────────────────────────┐
   │ 阶段 B: 方案审查（只有澄清通过后才执行） │
   │ → 根因验证、过度设计、成本审计、副作用   │
   └─────────────────────────────────────────┘
4. Subagent 返回会诊报告
5. 主 Agent 将会诊报告呈现给用户
6. 主 Agent 必须输出：
   - 原计划（未修订）
   - 会诊建议后的修订计划（如有）
7. **等待用户选择**（原计划 / 修订计划 / 取消）
```

## 输出格式

## Critique Subagent 审查清单

### 阶段 A: 需求澄清检查（优先执行，有问题则暂停）

| 检查项 | 检查内容 | 发现问题时的行动 |
|--------|----------|------------------|
| 🤔 需求合理性 | 用户要做的事情合理吗？有没有更好的方式？ | 提出替代建议 |
| ❓ 需求歧义 | 需求是否有多种理解方式？ | 列出歧义点，要求澄清 |
| 📋 信息完整性 | 是否缺少关键信息（边界条件、优先级、约束等）？ | 列出缺失信息，要求补充 |
| 🎯 目标明确性 | 用户真正想解决的问题是什么？表面需求 vs 真实需求？ | 追问真实意图 |
| ⚠️ 风险预警 | 用户是否意识到这个需求的潜在风险/成本？ | 提醒风险，要求确认 |

**⛔ 阶段 A 有任何问题 → 必须暂停，先向用户澄清，不进入阶段 B**

### 阶段 B: 方案审查（需求澄清通过后执行）

| 检查项 | 检查内容 |
|--------|----------|
| 🔬 根因验证 | 确认了还是在猜？基础检查做了吗？ |
| 🎯 过度设计 | 有没有 PM 没要求的东西？ |
| 💰 成本审计 | 有免费替代吗？根因确认了才花钱？ |
| ⚡ 副作用 | 会引入新问题吗？可回退吗？ |
| 🛡️ 保留性 | 用户说"添加"时是否做成了"替换"？同级/内层元素是否被意外删除？ |
| 🔗 关联性 | 是否有对称操作被遗漏？（编码改了，解码改了吗？）完整流程是否都检查了？ |
| 📐 继承性 | 改了基类/接口，子类/实现是否需要同步？ |
| 🧱 代码结构 | 是否在单文件堆积多种职责？新功能是否应融入现有模块而非新建？模块划分是否合理？ |

## Subagent 位置

| 位置 | 路径 | 用途 |
|------|------|------|
| 项目级 | `.cursor/agents/critique.md` | 仓库可提供或本地补充，优先用于项目上下文 |
| 用户级 | `~/.cursor/agents/critique.md` | 跟着电脑，通用版 |

优先使用项目级，因为可以理解项目业务背景。

## ⛔ 禁止行为

- 主 Agent 自己切换角色审查（有 bias）
- 跳过会诊直接执行方案
- 会诊时传递主 Agent 的思考过程
- **需求有歧义时直接开始方案审查**（必须先澄清）
- **发现澄清问题后仍输出方案选项**（必须等用户回答）

## 灵活性原则

Critique Subagent 必须：
1. 先读取 `.autodev/context-snapshot.md` 理解业务背景
2. 理解"弯弯绕绕"有时是业务需要
3. 有合理解释的方案可以接受
4. 目标是帮助决策，不是阻止执行

## 需求澄清的判断标准

**需要澄清的情况**：
- 用户说"做一个 XX 功能"但没说具体要求
- 用户的需求可能有多种实现方式，选择会影响后续
- 用户描述的问题现象，但根因不明确
- 用户要求的改动可能有副作用，但用户可能没意识到
- 需求涉及敏感操作（删除数据、修改权限、影响线上）

**不需要澄清的情况**：
- 用户需求已经足够具体和明确
- 用户明确说"不用问我，直接做"
- 细节可以由 AI 合理推断且用户可以在验收时调整
- 澄清问题的价值远小于直接执行再调整

## 模式集成（各模式会诊的共用入口）

各写模式在方案输出后，统一按本节执行会诊。各模式 README 只需写：

```
⭐ 方案输出后，按 `references/principles/critique.md` 的模式集成流程执行自动会诊。
本模式审查重点：[见下表]
```

### 各模式审查重点

| 模式 | 传递内容 | 阶段 A 澄清重点 | 阶段 B 审查重点 |
|------|----------|------------------|-----------------|
| **Architect** | 用户原始需求 + 开发方案 | 需求是否合理？有更好方式？缺关键信息？ | 过度设计？成本？副作用？ |
| **Refactor** | 用户重构目标 + 重构方案 | 重构目标是否明确？范围是否需确认？隐含业务约束？ | 是最简方案吗？会不会改太多？更安全的方式？ |
| **Optimize** | 用户优化诉求 + 优化方案 | "慢"是什么场景？有量化数据？优化目标明确？ | 过早优化？更简单方案？会影响功能？ |
| **Debug** | 用户问题描述 + 确诊结论 + 治疗方案 | 症状描述是否完整？需更多复现信息？ | 根因是否确认？是否对症？副作用？ |
| **Hotfix** | 用户问题描述 + 最小修复方案 | 问题描述是否足够定位？（快速版，最多 1-2 个问题） | 是否真的是最小改动？会引入新问题？ |
| **FastTrack** | 用户原始请求 + 快速方案 | 请求是否明确？（快速版，最多 1 个问题） | 是否真的是小改动？方案是否正确？ |

### 会诊后的通用输出

会诊完成后，所有模式统一使用以下格式（已在上方"输出格式"章节定义）：
- 情况 1：需求需要澄清 → 暂停，问用户
- 情况 2：需求无歧义 → 原计划 + 会诊报告 + 修订计划 + 用户选择

### 通用禁止规则

以下规则适用于所有使用会诊的模式：

- ⛔ 禁止跳过会诊直接让用户选择执行
- ⛔ 禁止会诊后直接执行，必须等待用户选择
- ⛔ 禁止在需求有澄清问题时输出方案选项
