# 测试编排示例

> 本文档展示 V1 测试协议在不同场景下的应用方式。

## 场景 1：后端小 Bug，前端页面不变

```text
任务类型: Debug
测试等级: 小测试
触发前端链路测试: 否（如未改接口契约且不命中关键用户链路）
```

执行方式：

1. 影响范围分析，确认改动点、调用方、返回结构是否变化。
2. 组装场景矩阵：
   - 原始复现路径
   - 一个错误输入
   - 一个边界 case
3. 默认执行后台自动测试：
   - 单测 / 集成测 / API smoke
4. 输出 `🧾 测试回执`。

## 场景 2：前几步先做后端，后几步再做前端

```text
任务类型: Step（非信任模式）
测试等级: 视链路而定，通常是大测试
触发前端链路测试: 最终联调步触发
```

执行方式：

1. Step 1-2 先做后端时，只验证当前已完成层级。
2. 回执里明确写：
   - 后台自动测试已通过
   - 前端链路“暂不可执行，等待后续步骤接通”
3. 到前后端联调步时，再提示是否执行 Playwright。

## 场景 3：登录 / 支付 / 权限等大功能

```text
任务类型: Architect -> Step
测试等级: 大测试
触发前端链路测试: 是
```

执行方式：

1. 在 Architect 阶段生成行为场景包。
2. 初始化 `.autodev/current-test.md`。
3. 每步默认跑后台自动测试。
4. 联调后和收尾前，主动提示执行 Playwright。
5. `current-test.md` 持续记录：
   - 场景矩阵
   - 待业务确认问题
   - 执行记录
   - 剩余风险

## 场景 4：PM 临时想到一个 use case，要在已有项目里测一下

```text
任务类型: Tester
测试等级: 小测试 / 大测试（视场景而定）
```

执行方式：

1. 先把自然语言翻译成行为场景。
2. 搜索项目里是否已有覆盖。
3. 没有覆盖时，判断：
   - 临时验证一次
   - 或补成长期回归测试
4. 若场景涉及业务阈值或权限矩阵，先向用户提问。

## 场景 5：前后端页面小功能

```text
任务类型: Architect
测试等级: 通常是小测试；命中关键链路时升级为大测试
```

执行方式：

1. 先生成行为场景包：
   - 用户提出的场景
   - AI 补充的负例 / 边界例
2. 后端部分默认补后台自动测试。
3. 页面接通后，AI 主动提示是否执行前端冒烟测试。
4. 若用户暂不执行，回执里明确标记“未执行”。
