# 标准比喻模板：餐厅 / 做饭

## 适用场景

- PM / 业务沟通
- 解释请求处理和结果交付
- 解释“系统内部好了，但用户还没看到”
- 解释前后端协作

## 不适用场景

- 高精度接口定义
- 底层并发细节
- 安全 / 支付 / 权限的严肃规则，只能辅助说明，不能替代技术解释

## 核心映射

| 技术对象 | 餐厅比喻 |
|----------|----------|
| 用户请求 | 点单 |
| Brainstorm | 确认吃什么、忌口和标准 |
| Architect | 排菜单和出餐顺序 |
| Step | 一道一道做菜 |
| Backend | 后厨 |
| Frontend | 上菜到桌 |
| Database | 仓库 / 冷库 |
| Queue / Async | 待做单队列 |
| Auth / Permission | 验券 / 会员核验 |
| Cache | 备菜台 |
| Test / Validation | 试吃 / 质检 |
| Release / 验收 | 试营业检查 |

## 常见故障映射

- 菜做好了但没上桌
  - 技术含义：结果已经生成，但没有传到用户可见层
- 点单没传进后厨
  - 技术含义：入口事件没有触发后端主链路
- 备菜台和后厨不同步
  - 技术含义：缓存 / 中间状态与真实数据不一致

## 用户常见问法

- 为什么没有上菜？
- 是不是点单没进后厨？
- 让一个专门上菜的人处理这一步可以吗？

## 推荐输出风格

- 先用一句餐厅比喻解释当前状态
- 再用一句技术结论兜底
